# UI/UX Design Guidelines

## 1. 目标

本文档用于规范当前 Shopify App 的整体视觉和交互表现，解决以下几个已出现的问题：

- 字号层级不统一，标题、正文、说明文和按钮文案混用 `12px / 14px / 16px / 18px / 20px / 24px / 28px`
- 颜色来源分散，Polaris token、Ant Design token、硬编码十六进制颜色同时存在
- 卡片、表单、状态标签、按钮在不同页面的内边距、圆角、边框和层级表达不一致
- 页面布局密度不统一，同一页面内存在“信息区很紧 + 卡片区很松”的情况
- 交互缺少统一规范，例如空状态、加载态、禁用态、表单帮助信息、保存反馈等表现不一致

本规范基于以下原则制定：

- 以 Shopify Polaris 作为视觉基准，保证嵌入式 App 与 Shopify Admin 的体验一致
- 保留现有 `Polaris + Ant Design` 的技术栈，但让 Polaris token 成为唯一的视觉源
- 优先通过设计规范收敛风格，再逐步治理散落在页面和组件里的硬编码样式

## 2. 当前问题诊断

### 2.1 主题入口分散

当前项目的主题配置散落在多个入口中：

- `app/entry.client.tsx`
- `app/entry.server.tsx`
- `app/routes/app.tsx`

目前只对 Ant Design 做了少量 token 配置，例如：

- `colorPrimary`
- `fontSize`
- `Table.rowSelectedBg`
- `Select.optionSelectedBg`

问题在于：

- 全局主题配置重复，后续维护成本高
- `fontSize: 16` 被作为全局默认值，但页面中大量文本仍然手写 `12px/14px/18px/20px`
- Ant Design 组件已经部分映射到 Polaris 颜色，但没有形成完整 token 体系

### 2.2 字号层级混乱

当前页面存在明显的字体层级混用：

- Dashboard 卡片标题常用 `14px`
- 次级按钮或辅助信息大量使用 `12px`
- 个别卡片标题直接使用 `20px`
- Pricing 页面核心数字使用 `24px/28px`
- 富文本相关区域又单独维护一套标题级别

结果是：

- 页面“视觉重心”不稳定
- 标题与正文的对比不够精确
- 信息密度高的页面可读性下降

### 2.3 颜色不专业且语义不统一

当前颜色大量直接写死在 CSS 或内联样式中，例如：

- `#007F61`
- `#168c6d`
- `#999`
- `#d9d9d9`
- `#4caf50`
- `#2196f3`
- `#faad14`
- `#ff4d4f`
- `#000`

问题在于：

- 相同语义没有共用同一组 token
- 中性色、品牌色、状态色没有固定层级
- 某些页面偏“运营后台风格”，某些页面偏“营销风格”，专业感被削弱

### 2.4 布局和间距缺少统一节奏

当前常见的间距写法包括：

- `12px 24px`
- `8px 12px`
- `4px 8px`
- `2px 10px`
- `10px`
- `20px`

常见问题：

- 同一页面内卡片 body padding 不统一
- 表单项、操作区、列表区的垂直节奏不稳定
- Switcher 编辑页和 Dashboard 首页的密度差异过大

### 2.5 交互模式缺少统一定义

目前缺少系统规范的部分包括：

- 页面主操作与次操作的优先级表达
- 表单帮助信息和错误反馈的固定位置
- 表格空态、加载态、无权限态的统一结构
- 状态标签的颜色、形状、文本风格不一致
- Dashboard 卡片中 CTA 的位置和样式不稳定

## 3. Shopify App 设计原则

结合 Shopify App 与 Polaris 的通用规范，当前项目建议采用以下设计原则：

### 3.1 与 Shopify Admin 保持一致

- 页面基础背景、卡片表面、边框、文本层级优先使用 Polaris token
- 不再单独发明新的“主色体系”，品牌强调色只用于最关键的主操作
- 每个区域只允许一个强主按钮，避免多个高饱和按钮同时争抢注意力

### 3.2 让信息层级清晰，而不是让颜色更重

- 标题、正文、辅助文本的区分主要依赖字号、字重和间距
- 状态信息使用语义色，不使用装饰性色块
- 数据卡片优先通过布局和数字层级突出重点，而不是依赖高饱和背景

### 3.3 同一页面保持一致的信息密度

- 一个页面只选一种主密度，不要同页混用松散卡片和超紧凑操作块
- Dashboard 采用中密度
- Settings 页面采用中低密度
- 表格和资源管理页采用中高密度

### 3.4 交互应当让商家“可预期”

- 所有表单页应有固定的保存反馈机制
- 所有批量操作都要明确显示作用对象、进行中状态和完成反馈
- 所有空态都应解释“为什么为空”和“下一步做什么”

## 4. 设计系统方案

### 4.1 设计系统策略

建议采用以下策略：

- `Polaris` 作为视觉与语义 token 的唯一来源
- `Ant Design` 继续用于表格、图表、复杂表单和弹窗能力
- 所有 Ant Design 主题覆盖都从 Polaris token 派生，不再直接写十六进制颜色
- 页面和组件不再新增随意的内联颜色/字号/圆角

一句话原则：

> 视觉标准来自 Polaris，交互容器和业务复杂组件可以继续使用 Ant Design。

### 4.2 推荐基础 token

以下 token 不要求立刻抽成代码常量，但必须作为统一标准执行。

#### 颜色

优先使用 Polaris CSS variables：

- 页面背景：`var(--p-color-bg)`
- 一级卡片背景：`var(--p-color-bg-surface)`
- 二级背景：`var(--p-color-bg-surface-secondary)`
- 悬浮背景：`var(--p-color-bg-surface-hover)`
- 选中背景：`var(--p-color-bg-surface-selected)`
- 边框：`var(--p-color-border)`
- 次级边框：`var(--p-color-border-secondary)`
- 主文本：`var(--p-color-text)`
- 次文本：`var(--p-color-text-secondary)`
- 弱化文本：`var(--p-color-text-tertiary)`
- 品牌主操作：`var(--p-color-bg-fill-brand)`
- 主操作文字：`var(--p-color-text-brand-on-bg-fill)`
- 成功：`var(--p-color-bg-surface-success)` + `var(--p-color-text-success)`
- 提醒：`var(--p-color-bg-surface-caution)` + `var(--p-color-text-caution)`
- 危险：`var(--p-color-bg-surface-critical)` + `var(--p-color-text-critical)`
- 信息：`var(--p-color-bg-surface-info)` + `var(--p-color-text-info)`

补充限制：

- 常规业务页面不要使用纯黑 `#000` 作为标题色、表头色、输入框边框色或按钮背景
- 文字层级优先用 `var(--p-color-text)`、`var(--p-color-text-secondary)`、`var(--p-color-text-tertiary)` 区分
- 输入框、选择器、表格表头都应保持中性灰阶，不要为了强调而变成纯黑
- 只有代码、模板变量、开发者信息等极少数场景才允许使用更强对比的深色表达

禁止新增以下模式：

- 直接写 `#999` 作为全局次级文本
- 直接写 `#000` 作为按钮背景
- 为每个状态手工挑一组近似颜色

#### 字体

基础字体遵循 Shopify Admin 方向：

- 英文与数字：优先使用系统栈或 Inter 风格
- 中文场景：允许退回系统字体，不额外引入装饰字体
- 代码、模板变量、API Key：使用等宽字体

#### 字号层级

建议收敛为以下 7 个常用层级：

| 场景 | 尺寸 | 字重 | 用法 |
| --- | --- | --- | --- |
| Page Title | 24px | 700 | 页面标题，仅页面头部使用 |
| Section Title | 20px | 600 | Dashboard 区块标题、大模块标题 |
| Card Title | 14px | 600 | 卡片标题、表单分组标题 |
| Body | 14px | 400 | 正文、说明、输入项主文案 |
| Body Small | 13px | 400 | 次级说明、表格次信息 |
| Caption | 12px | 400/500 | 状态补充、辅助信息、标签 |
| Micro | 11px | 400 | 时间戳、弱化元信息，仅极少场景使用 |

规则：

- 页面内默认正文统一为 `14px`
- 说明性文字优先用 `13px`，不要直接降到 `12px`
- `12px` 只保留给标签、状态补充、极弱辅助信息
- 避免在常规业务页大量使用 `16px` 作为正文

#### 行高

- `24px`：标题和 14px 正文的默认阅读行高
- `20px`：13px/12px 文案行高
- 数据类紧凑布局允许更小，但必须成组统一

#### 间距

统一采用 4px 基线网格，使用以下层级：

- `4`
- `8`
- `12`
- `16`
- `20`
- `24`
- `32`

推荐映射到 Polaris spacing token：

- `4px` -> `var(--p-space-100)`
- `8px` -> `var(--p-space-200)`
- `12px` -> `var(--p-space-300)`
- `16px` -> `var(--p-space-400)`
- `20px` -> `var(--p-space-500)`
- `24px` -> `var(--p-space-600)`

规则：

- 页面区块间距：`24px`
- 卡片内部主 padding：`16px`
- 信息密度较高的卡片可用 `12px 16px`
- 表单项之间垂直间距：`12px` 或 `16px`
- 行内控件 gap：`8px`

#### 圆角和阴影

- 默认圆角：`8px`
- 小型标签或轻量输入容器：`6px`
- 胶囊标签：9999px
- 卡片阴影尽量弱化
- 卡片默认不依赖明显边框做分隔，优先通过页面背景、留白、分组结构和极弱阴影拉开层级
- 如果必须有分隔，只允许使用非常轻的中性边界，不能出现“卡片描边感很强”的效果

## 5. 页面模板规范

### 5.1 Dashboard

适用页面：

- `app/routes/app._index/route.tsx`

建议结构：

1. 页面头部：标题 + 关键状态/计划信息
2. 关键数据摘要区：3 到 4 张同规格数据卡
3. 主要操作区：翻译入口、语言入口、切换器入口
4. 支持与引导区：帮助文档、客服、引导信息

规范：

- 页面标题和卡片标题分层明确，不要让多个 `Title level={4}` 同时承担不同层级
- 首页中的卡片标题统一使用 `14px/600`
- 统计数值统一使用数字样式，不要混用粗体正文和标题字号
- 所有 CTA 在卡片底部对齐，避免有的靠左、有的靠右、有的穿插在正文中

### 5.2 Settings 页面

适用页面：

- `app/routes/app.language/route.tsx`
- `app/routes/app.currency/route.tsx`
- `app/routes/app.apikeySetting/route.tsx`
- `app/routes/app.glossary/route.tsx`

建议结构：

1. 页面头部：标题 + 页面说明
2. 主配置区：按配置分组分卡展示
3. 二级说明区：帮助链接、风险提示、文档入口

规范：

- 每张设置卡只讲一类事情
- 卡片头部包含标题和一句解释，不要让正文第一行承担“标题补充”的角色
- 表单帮助信息固定放在控件下方
- 错误信息固定放在控件下方，不要和按钮并排出现

### 5.3 资源管理页

适用页面：

- `app/routes/app.manage_translation*.tsx`

建议结构：

1. 头部筛选区
2. 批量操作区
3. 表格主体
4. 详情抽屉/弹窗

规范：

- 资源管理页优先使用 full-width 布局
- 表头筛选、搜索、批量操作的高度统一
- 表格状态色统一改为 token 驱动
- 批量操作只保留一个主按钮，其余为次按钮或文本操作

### 5.4 Switcher 编辑页

适用页面：

- `app/routes/app.switcher/route.tsx`

该页面最适合 Shopify 推荐的双栏编辑模式：

- 左侧：设置面板
- 右侧：实时预览

规范：

- 桌面端保持稳定的双栏比例，不要出现编辑区和预览区风格断裂
- 左侧配置项按 section 分组，section 间距统一
- 右侧预览容器本身也要遵循同一套 token，不再用独立黑色按钮风格
- 移动端折叠为单栏时，预览区应位于设置区之后，并有明确标题

### 5.5 Pricing 页面

适用页面：

- `app/routes/app.pricing/route.tsx`

Pricing 页面当前视觉最容易偏离 Shopify Admin 风格，需要重点收敛：

- 避免营销着陆页式的大面积强色块
- 优先使用卡片对比，而不是高饱和背景对比
- 价格数字可以突出，但标题、副标题、功能列表应回到后台应用的冷静层级
- “当前计划”“推荐计划”“购买额度”三种信息必须明确分层

但 Pricing 页面也不能过于“后台说明页化”，否则会直接削弱转化动机。允许做有限度的营销强化，前提是仍然保持 Shopify Admin 的克制感。

推荐结构：

1. 页面头部：标题 + 一句价值说明 + 可选的次级入口
2. 价值摘要区：轻渐变或轻强调背景的 Hero Card，说明购买计划的核心收益
3. 用量与加购区：当前额度、刷新时间、快捷加购
4. 套餐选择区：计费切换 + 套餐卡
5. 对比与 FAQ：降低决策摩擦

视觉规则：

- 允许在 Hero Card 和推荐套餐卡使用轻度渐变或品牌色染色，但透明度必须低，不能做满版深色块
- 允许价格数字比普通页面更强，但建议控制在 `32px - 36px` 区间，不要继续放大成落地页式夸张标题
- 推荐套餐可以使用更强阴影、轻色带、Ribbon 或 Tag 强化，但不能靠黑色标题栏或重描边制造对比
- 快捷加购区应优先表达“当前折扣已自动应用”，帮助用户快速理解为什么现在购买更划算
- 年付优惠必须在切换器附近重复出现，不能只藏在套餐副标题里
- 套餐卡内建议补充一句“适合谁”或“为什么升级”的摘要，避免卡片只剩价格和长功能列表
- FAQ 左侧可增加人工支持或选型建议入口，降低商家决策成本

文案规则：

- 标题偏价值表达，例如“让翻译预算更可预测”，而不是单纯“Choose the right plan”
- 副标题说明收益和场景，不堆砌抽象形容词
- CTA 文案优先清晰表达动作和结果，例如“Switch to yearly”“Browse all credit packs”
- 所有营销性文案仍应服务于“更好理解价格与权益”，而不是制造情绪化刺激

## 6. 组件规范

### 6.1 Page Header

组成：

- 页面标题
- 一句简短说明
- 右侧主操作或状态

规则：

- 一个页面头部最多一个主按钮
- 页面说明不超过两行
- 页面头部与正文首区块间距固定为 `16px`

### 6.2 Card

规则：

- 标准卡片：`padding: 16px`
- 紧凑卡片：`padding: 12px 16px`
- Card Title 统一 `14px/600`
- 卡片正文默认 `14px/400`
- 卡片底部操作区统一与正文保持 `12px` 间距
- 卡片默认无明显描边
- 卡片背景保持 `surface`，通过外层页面底色和区块间距形成层级

不建议：

- 卡片标题直接使用 `20px`
- 卡片正文中混入多个不同灰度的辅助文字
- 每张卡片各自定义一套 header/body padding
- 给卡片加明显边框来制造“存在感”

### 6.3 Button

规则：

- Primary：仅用于本区域最重要动作
- Secondary：普通操作
- Tertiary / Text：低优先级动作
- Destructive：删除、关闭、停用等不可逆操作

规范：

- 同一区域最多一个 Primary
- 不再使用纯黑按钮作为默认主按钮
- 文字按钮不要同时叠加高饱和文字色和粗体

### 6.4 Status Badge / Status Pill

适用状态：

- untranslated
- translating
- translated
- partly translated
- exception

统一样式建议：

- 采用浅底色 + 语义文本色 + 轻量图标/圆点
- 标签高度统一
- 文本字号统一为 `12px`
- 避免不同状态使用不一致的边框样式和动画逻辑

语义映射：

- Untranslated -> neutral
- Translating -> info
- Translated -> success
- Partly translated -> caution
- Exception -> critical

### 6.5 Table

规则：

- 表头、行高、选中态都由统一 token 控制
- 选中行背景不要使用单独的灰色常量，改为 Polaris neutral selected token
- 批量选择、筛选、分页区高度统一
- 单元格中的次级信息使用 `13px`，标签使用 `12px`
- 表格表头禁止使用纯黑背景或纯黑文字强化对比，应维持 Shopify Admin 风格的中性表头层级

### 6.6 Form

规则：

- 表单标签和帮助信息位置固定
- 必填、错误、警告和成功反馈的视觉样式固定
- 输入框、选择器、checkbox 容器的边框色和 hover 色全部统一
- 不再单独对某个卡片里的 checkbox 手动设计 hover 绿边框
- 表单标题、字段标签、placeholder、输入边框都不能突然跳成黑色
- 输入框默认应为浅边框 + 白底，focus 时使用品牌色或信息色的轻强调，而不是黑边框
- 表单分组标题应使用 `14px/600` 或 `13px/600` 的中性文本，不要用黑块、黑标题条或重色块分隔

### 6.7 Modal

规则：

- 标题层级统一
- 主操作按钮放右侧
- 次操作按钮为次级或文本按钮
- 危险操作必须配合说明文
- Modal 里的信息密度要比页面正文略紧，但不应比表格更紧

## 7. 交互规范

### 7.1 加载态

- 页面首次加载：优先使用 skeleton，不使用空白区域
- 卡片内局部请求：只替换该卡片内容，不阻塞整个页面
- 列表刷新：保留表头与布局，替换表格内容为 skeleton/placeholder

### 7.2 空状态

每个空状态都必须回答三件事：

- 当前为什么为空
- 用户下一步可以做什么
- 这个动作会带来什么结果

空状态结构建议：

- 图标或轻插画
- 1 行标题
- 1 到 2 行说明
- 1 个主操作

### 7.3 成功与错误反馈

- 表单保存成功：Toast
- 表单保存失败：字段级错误 + 顶部总结提示（必要时）
- 批量任务开始：明确提示任务已开始，并告知可在哪里查看进度
- 长任务执行中：页面内保持进度入口可见

### 7.4 禁用态

- 禁用不只改变透明度，还要让文案、图标和边框一起降级
- 禁用时必须给出原因说明，特别是套餐限制、未配置语言、额度不足等场景

## 8. 代码落地建议

### 8.1 第一阶段：收敛 token

优先处理：

- `app/routes/app.tsx`
- `app/entry.client.tsx`
- `app/entry.server.tsx`
- `app/styles.css`

动作：

- 建立单一主题入口
- 将 Ant Design token 改为从 Polaris CSS variable 派生
- 定义统一的字体层级、圆角、表格选中态、卡片 padding 规则

### 8.2 第二阶段：整改首页和高频页面

优先级：

1. Dashboard
2. Language
3. Translate
4. Manage Translation
5. Switcher
6. Pricing

动作：

- 去掉重复的内联 `fontSize`
- 收敛卡片标题、正文、辅助信息层级
- 统一 CTA 位置
- 替换硬编码状态色和边框色

### 8.3 第三阶段：建立公共样式基类或轻量封装

建议新增统一封装，而不是继续复制页面内联样式：

- `AppPageHeader`
- `AppSectionCard`
- `AppStatusBadge`
- `AppEmptyState`
- `AppFormSection`

这样可以避免同一套 UI 规则在每个页面重复实现。

## 9. 页面级整改重点

### Dashboard

- 减少首页中分散的小按钮
- 统一卡片标题和说明文案层级
- 支持信息、帮助入口降级，不与主任务抢视觉焦点
- 卡片之间通过留白分组，不通过边框堆叠层次

### Translate

- 语言选择卡、模型卡、术语表卡统一边框与选中态
- 当前 `hover` 绿色边框过于“局部定义”，应改为全局交互规范
- 复杂配置的展开/收起状态需要统一 chevron、标题和说明结构
- 输入框、选择器、复选容器要避免出现黑色边框和黑色标题条

### Manage Translation

- 资源管理页的筛选、状态、表格密度统一
- 状态标签与批量操作条统一为一套系统组件

### Switcher

- 预览区视觉必须回归后台应用风格，弱化独立站组件风格
- 黑色按钮和独立颜色体系需要替换
- 桌面和移动端的预览切换逻辑需要更稳定的结构层次

### Pricing

- 控制营销感，而不是完全取消营销感
- 允许在 Hero、推荐卡、年付优惠上做轻度强调，但保持低饱和和后台质感
- 强化套餐对比的可读性和可信度
- 用中性卡片 + 清晰主按钮 + 有解释力的摘要文案替代纯视觉刺激
- 套餐卡可以突出，但不要靠黑色标题栏或明显描边制造对比

## 10. Do / Don't

### Do

- 使用 Polaris token 作为颜色和间距源
- 将正文统一回到 `14px`
- 让 `12px` 只服务于标签和弱化信息
- 让卡片标题固定为 `14px/600`
- 同一区域只保留一个主按钮
- 让状态颜色只表达语义，不承担装饰作用
- 让卡片主要通过背景、留白和结构分组体现层级
- 让表单保持浅色、克制、中性的后台风格

### Don't

- 不要在页面里继续新增 `style={{ fontSize: "12px" }}` 这类零散定义
- 不要用 `#000` 做默认主按钮背景
- 不要在不同页面定义不同的选中灰、提示绿、说明灰
- 不要让同一页同时出现多种卡片 padding 标准
- 不要把“帮助信息”“主说明”“状态提示”都做成同一种视觉强度
- 不要给常规卡片加明显描边
- 不要让表头、分组标题、输入边框突然变成黑色

## 11. 验收标准

当以下条件满足时，可以认为整体 UI 进入统一状态：

- 页面标题、区块标题、卡片标题、正文、辅助文案均可归入固定字级
- 主色、状态色、中性色全部来自统一 token
- 80% 以上卡片使用统一 padding 和圆角
- 高流量页面不再出现明显的局部手工配色
- 新页面可以直接复用 Page Header / Section Card / Status Badge 等规范

## 12. 结论

当前项目的问题不是“某一个页面不够美观”，而是缺少一套统一的后台应用设计系统。

最优解不是单独改字号或换几组颜色，而是：

1. 明确以 Polaris 为视觉基准
2. 用统一 token 约束 Ant Design 和自定义样式
3. 先改首页、设置页、资源管理页，再逐步替换散落的硬编码样式

如果后续进入执行阶段，建议下一步先从 `Dashboard + Translate + Switcher` 三个页面开始做第一轮视觉收敛。
