# UI Refactor Audit

## 1. 文档定位

本文档用于记录当前 Shopify App 的 UI 重构审查结果，重点描述以下内容：

- 当前页面结构与模块分组
- 已确认的 UI/交互问题
- 与 `design.md` 之间的职责边界

本文档**不包含执行计划、排期、任务拆分或改造步骤**。  
`design.md` 继续作为视觉与交互规范文档使用；本文件只负责记录“当前实现有哪些问题，需要重构什么”。

---

## 2. 与 `design.md` 的边界

### `design.md` 负责

- 设计原则
- 视觉规范
- 交互规范
- token 约束
- 页面模板标准
- 组件标准

### 本文档负责

- 当前项目 UI 结构盘点
- 页面级问题审查
- 与规范不一致的具体现象
- 后续进入重构前需要确认的页面问题清单

## 3. 当前 UI 结构总览

当前项目可以按以下层级理解：

### 3.1 全局层

负责应用壳、Polaris/Ant Design 组合方式、导航和全局样式入口：

- `app/routes/app.tsx`
- `app/entry.client.tsx`
- `app/entry.server.tsx`
- `app/styles.css`

这一层的主要问题是主题入口分散、token 覆盖不集中、Ant Design 与 Polaris 的映射仍不完整。

### 3.2 页面层分组

#### Dashboard

- `app/routes/app._index/route.tsx`

承担首页数据摘要、翻译任务入口、任务进度、Switcher 引导、帮助信息等内容。

#### Settings 类页面

- `app/routes/app.language/route.tsx`
- `app/routes/app.currency/route.tsx`
- `app/routes/app.apikeySetting/route.tsx`
- `app/routes/app.glossary/route.tsx`

这类页面以配置和表单为主，后续重点关注卡片分组、表单帮助信息、反馈与状态一致性。

#### Translate 配置页

- `app/routes/app.translate/route.tsx`

承担翻译语言、模型、术语表、附加设置等核心配置。

#### Manage Translation 资源管理页

- `app/routes/app.manage_translation/route.tsx`
- `app/routes/app.manage_translation_*`

承担筛选、批量操作、资源列表和详情操作，是高密度业务页。

#### Switcher 编辑页

- `app/routes/app.switcher/route.tsx`

承担双栏配置与预览，是视觉偏差较明显的页面之一。

#### Pricing 页面

- `app/routes/app.pricing/route.tsx`

承担套餐、额度和购买相关信息，是最容易偏离 Shopify Admin 风格的页面之一。

## 4. 当前全局 UI 实现问题

### 4.1 主题入口仍然分散

当前主题配置分散在多个入口中：

- `app/entry.client.tsx`
- `app/entry.server.tsx`
- `app/routes/app.tsx`

主要问题：

- Ant Design 全局 token 配置存在重复定义
- 路由壳层又追加了一层主题覆盖
- 页面内仍然存在大量独立内联样式，导致“有主题但没有真正统一”

### 4.2 Polaris 与 Ant Design 尚未形成单一视觉源

虽然项目已经明确采用 `Polaris + Ant Design` 组合，但当前并未完全做到“视觉标准统一来自 Polaris”：

- Ant Design 只做了少量 token 覆盖
- 很多页面仍然直接写十六进制颜色
- 表格、按钮、卡片、状态色尚未完全映射到 Polaris 语义 token

### 4.3 全局样式层过薄，页面局部样式过重

当前 `app/styles.css` 内容较少，更多样式控制散落在页面组件里：

- 内联 `style={{}}`
- `styles.css`
- `styles.module.css`

结果是：

- 难以形成统一的页面密度
- 难以批量收敛标题、正文、间距、圆角和边框
- 后续重构成本被放大

## 5. 页面级审查

### 5.1 Dashboard

涉及文件：

- `app/routes/app._index/route.tsx`
- `app/routes/app._index/components/AnalyticsCard.tsx`
- `app/routes/app._index/components/TranslationPanel.tsx`
- `app/routes/app._index/components/welcomeCard.tsx`
- `app/routes/app._index/components/progressingCard.tsx`

#### 页面当前结构

当前首页可以拆为以下几个区域：

1. 页面顶部：`TitleBar + ScrollNotice`
2. 数据摘要区：`AnalyticsCard`
3. 任务进度区：`ProgressingCard`
4. 主要操作区：`TranslationPanel`
5. Switcher 引导区：`WelcomeCard`
6. 支持与说明区：说明卡片、roadmap 表格、帮助外链

#### 已确认问题

##### 1. 缺少统一的页面头部结构

当前首页顶部只有 `TitleBar` 和滚动通知，没有形成固定的：

- 页面标题
- 一句说明
- 右侧主操作或状态

这导致首页进入后视觉焦点不稳定，也不符合 `design.md` 中的 `Page Header` 规范。

##### 2. 标题层级混乱

同一页同时混用了多种标题表达方式：

- `20px` 卡片标题
- `Title level={4}`
- `Text strong`
- 手写 `14px`

结果是：

- 页面标题、区块标题、卡片标题边界不清
- 卡片之间视觉权重不一致
- 首页信息结构不够稳定

##### 3. 间距体系不统一

首页中同时出现了以下典型写法：

- `12px 24px`
- `8px`
- `10px`
- `20px`
- `14px`

这与 `design.md` 中推荐的 4px 基线和 `16px / 24px` 主节奏不一致，导致页面密度忽紧忽松。

##### 4. CTA 优先级不清晰

首页存在多个按钮入口，但没有统一的主次规则：

- 有的区块用主按钮
- 有的区块用默认按钮
- 有的区块用文字链接
- 有的区块把多个功能入口并排展示

结果是多个动作同时争抢注意力，不符合“同一区域最多一个 Primary”的设计原则。

##### 5. 颜色来源不统一

首页仍存在大量硬编码颜色，例如：

- `#007F61`
- `#999`
- `#333`
- `#52c41a`
- `#faad14`
- `#ff4d4f`

这些颜色没有统一通过 Polaris token 派生，语义与视觉强度也不一致。

##### 6. 主要操作区信息架构不够清晰

`TranslationPanel` 当前把以下不同类型的任务并列呈现：

- 添加语言与自动翻译
- 翻译资源管理
- 术语表与其他工具

虽然都属于翻译相关能力，但任务性质并不相同。现在的呈现方式会让页面显得按钮密度偏高，且不利于用户快速判断下一步。

##### 7. 支持信息权重偏高

首页下半部分同时包含：

- 说明卡片
- 帮助卡片
- roadmap 表格
- 外链帮助入口

这些内容都属于支持与引导信息，但当前权重偏高，容易与首页主任务竞争视觉焦点。

##### 8. 交互反馈模式还不统一

首页不同区块分别使用了：

- Skeleton
- Modal
- 默认按钮 loading
- 普通文本空态

这些反馈方式各自可用，但还没有形成统一的页面级交互规则。

##### 9. 内联样式过多

Dashboard 及其子组件中存在大量内联样式，导致：

- 视觉规范难以统一下沉
- token 替换难以批量完成
- 后续 UI 优化会持续依赖“局部手工修补”

#### 当前结论

Dashboard 当前最核心的问题不是某一张卡片不好看，而是：

- 页面头部缺少统一结构
- 信息层级没有稳定下来
- CTA 优先级不清晰
- 颜色、间距和标题未完全回到统一规范

这意味着首页在进入正式重构前，应该首先被视为“结构与规范一致性问题”，而不是“局部样式微调问题”。

---

## 6. 当前审查结论

从目前已确认的内容来看，当前项目的 UI 问题主要集中在三层：

### 6.1 全局层

- 主题入口分散
- token 收口不足
- Polaris 与 Ant Design 的映射不完整

### 6.2 页面层

- 页面结构分区不稳定
- 同一页面内的信息密度不统一
- 帮助信息、主操作、状态信息经常处于相近视觉强度

### 6.3 组件层

- 卡片标题标准不一致
- padding 和间距标准不一致
- 状态色和边框色缺少统一来源
- 大量内联样式使组件难以复用

---

## 6. 页面级补充审查

### 6.1 Translate

涉及文件：

- `app/routes/app.translate/route.tsx`
- `app/routes/app.translate/styles.module.css`
- `app/routes/app.translate/components/languageSelectorCard.tsx`
- `app/routes/app.translate/components/transalteSettingCard.tsx`
- `app/routes/app.translate/components/toneSettingCard.tsx`
- `app/routes/app.translate/components/advanceSettingCard.tsx`
- `app/routes/app.translate/components/translateAffix.tsx`

#### 页面当前结构

当前 Translate 页面可以拆为以下几个区域：

1. 页面顶部吸附操作条：返回按钮 + 页面标题 + 主操作按钮
2. 页面提示区：`ScrollNotice`
3. 语言选择区：目标语言选择卡片
4. 翻译设置区：翻译范围、翻译模式、模型选择
5. 风格设置区：翻译语气、品牌风格、CTA 强度、文案格式、行业分类
6. 高级设置区：术语表等附加能力
7. 弹窗反馈区：首翻提醒、额度提醒、功能限制提醒

#### 已确认问题

##### 1. 页面头部结构不稳定

虽然页面有 `Affix` 吸附操作条，但当前头部是手工用 `div + Button + Title` 拼出来的，且背景直接写为：

- `rgb(241, 241, 241)`

这意味着 Translate 页虽然已经有“页面头部”的雏形，但没有真正纳入统一的 `Page Header` 体系，也没有回到 Polaris token 驱动的背景和边框语义。

##### 2. 标题层级与区块标题标准不统一

页面主标题和多个 section 标题都使用了手写样式，例如：

- `fontSize: "1.25rem"`
- `fontWeight: 700`
- 多处 `Title level={5} + fontSize: "1rem"`

问题在于：

- 页面标题、区块标题、卡片标题没有落到固定字级
- 多个 section 的标题视觉表现接近，但语义层级并不一致
- 页面整体更像“很多局部标题堆叠”，而不是一个层级明确的设置页

##### 3. 配置卡片分组虽然明确，但卡片规范还未统一

Translate 页已经具备按业务分组的基础，这是优点；但当前各卡片的表现仍不统一：

- 有的卡片直接用 `Card`
- 有的卡片自带 `title`
- 有的标题写在卡片外部
- 有的区域通过 `Divider` 人工切开

结果是：

- 页面结构可以理解，但视觉上还没有形成统一的“设置页模板”
- 标题和内容的关系不够稳定
- 很多卡片像是功能拼接，而不是同一套设计系统下的设置面板

##### 4. 颜色与交互边框仍然存在局部手工定义

`Translate` 页当前仍有明显的硬编码交互样式，例如：

- hover 绿色边框 `#168c6d`
- 默认边框 `#f0f0f0`
- 默认边框 `#d9d9d9`
- 校验态直接把语言卡片边框改成红色
- 旗帜边框使用 `#888`

这与 `design.md` 中“选中态、hover、状态色统一由 token 驱动”的原则不一致，也会导致局部交互手感和全局不一致。

##### 5. 语言选择区交互方式偏手工实现

语言选择卡片当前通过以下方式实现交互：

- `Checkbox.Group`
- 自定义网格布局
- 通过 DOM ref 直接修改边框颜色
- 通过错误文案补充校验反馈

当前问题不是功能不可用，而是：

- 表单错误反馈没有遵循统一的字段级结构
- 选中态、错误态、hover 态都比较依赖局部样式控制
- 语言卡片看起来像“自定义选择器”，但没有被抽象成统一的可复用系统组件

##### 6. 页面正文密度略高，信息节奏偏紧

当前页面纵向堆叠了很多功能区：

- 语言选择
- 模型选择
- 翻译范围
- 风格与行业设置
- 高级设置

但页面里很多标题、说明和控件之间的节奏仍然比较依赖局部 margin、padding 和 `Space` 组合，没有完全形成统一的设置页阅读节奏。这会让页面在信息较多时显得偏紧。

##### 7. 风格设置区内容丰富，但信息表达偏“配置清单”

`ToneSettingCard` 提供了很多能力：

- 语气
- 品牌风格
- CTA 强度
- 文案格式
- 行业类别

能力本身是完整的，但当前问题在于：

- 各字段的解释方式几乎一致，没有形成主次关系
- 所有配置都以相似强度展示，阅读成本偏高
- 大量选项文案较长，视觉上容易显得冗长

这会让用户感觉“可配很多”，但不一定能快速理解“哪些配置最重要”。

##### 8. 高级设置区折叠逻辑有了，但呈现仍然较轻量和零散

`AdvanceSettingCard` 现在只放了一个 Glossary 开关，并通过折叠按钮控制展开。  
问题在于：

- 卡片标题仍然使用默认写法，没有和其他 section 对齐
- 展开前后卡片密度变化明显
- 当前只包含一个开关时，折叠容器显得有些“结构重于内容”

这意味着高级设置区后续应继续收敛为更稳定的 section 模式，而不是继续叠加零散开关。

##### 9. 模态反馈较多，但反馈体系尚未完全统一

当前 Translate 页里至少有以下几类反馈：

- 首次翻译提示
- 私有额度不足提示
- 功能不可用提示
- 支付提示
- Toast 提示

这些反馈分别解决不同问题，但当前仍存在以下现象：

- 成功、失败、限制、引导分散在不同反馈组件中
- 字段错误、顶部提示、弹窗提示的边界还不够统一
- 页面级反馈模型尚未沉淀为稳定规范

##### 10. 页面存在较多内联样式与局部布局逻辑

虽然 Translate 页已经比部分页面更接近“分组件结构”，但内部仍然存在大量：

- 行内 `style`
- 手工 `paddingLeft`
- 局部 `gridTemplateColumns`
- 手工写死的卡片边框和圆角

这会直接影响后续基于 token 的统一替换和复用。

#### 当前结论

Translate 页相比 Dashboard 已经更接近“设置页”结构，但它当前的主要问题在于：

- 设置分组有了，规范统一性还不够
- 交互状态有了，视觉语义还没有统一来源
- 配置项很多，但主次节奏和说明层级不够清楚

因此，这一页后续重构重点不应是简单“换皮”，而应是把现有分组结构进一步收敛到统一的设置页模板中，让标题、卡片、帮助信息、错误反馈和交互状态全部回到同一套规则。

---

### 6.2 Switcher

涉及文件：

- `app/routes/app.switcher/route.tsx`
- `app/routes/app.switcher/styles.module.css`
- `app/routes/app.switcher/components/switcherSettingCard.tsx`

#### 页面当前结构

当前 Switcher 页面可以拆为以下几个区域：

1. 页面顶部：`TitleBar + ScrollNotice`
2. 引导卡片区：`SwitcherSettingCard`
3. 左侧配置区：自动切换、显示模式、样式配置、位置配置
4. 右侧预览区：模拟语言/货币切换器的实时预览
5. 底部反馈区：功能限制弹窗

整体上，这一页已经具备“左侧设置 + 右侧预览”的编辑器雏形，但实现方式和视觉风格明显偏离 `design.md` 中推荐的 Shopify 后台应用双栏编辑模式。

#### 已确认问题

##### 1. 双栏结构有了，但编辑器模板还不稳定

当前页面使用：

- 左侧配置
- 右侧预览

这一方向是对的，但当前布局只是简单的 `flex` 双栏，且主要依靠局部 CSS 和大量内联样式支撑。  
问题在于：

- 没有统一的编辑器页面模板
- 左右区域的标题、卡片节奏和区块间距不一致
- 桌面端与移动端切换虽然可用，但结构规则仍不够明确

##### 2. 页面头部仍未形成统一的编辑页 Header

目前顶部依然是：

- `TitleBar`
- `ScrollNotice`

而不是清晰的：

- 页面标题
- 页面说明
- 当前状态或保存相关信息

虽然页面通过 `SaveBar` 处理保存行为，但从视觉结构上看，用户并没有在页面头部获得明确的编辑上下文。

##### 3. 预览区视觉明显偏离 Shopify Admin 风格

这是当前 Switcher 页面最核心的问题之一。

预览区目前使用了大量独立配色和拟物化浏览器外观，例如：

- 灰色浏览器窗口背景
- 红黄绿窗口按钮
- 灰色容器边框
- 自定义弹层样式

这些表达方式更像“营销演示组件”或“独立站前台控件模拟器”，而不是一个 Shopify App 内部的后台编辑预览区。

这与 `design.md` 中“预览区应回归后台应用风格，弱化独立站组件风格”的要求不一致。

##### 4. 预览区存在独立的黑色按钮和自建视觉体系

`styles.module.css` 中存在明显不符合规范的样式，例如：

- 黑色主按钮 `#000`
- hover 深灰 `#333`
- 自定义滚动条颜色
- 自定义阴影、边框、圆角

这些样式构成了一套独立于 Polaris token 之外的局部视觉系统，属于 `design.md` 明确要收敛的内容。

##### 5. 颜色配置能力过于自由，但缺少系统约束

页面左侧允许用户直接配置：

- 字体颜色
- 背景颜色
- 边框颜色

这本身是业务能力，但在当前实现下存在两个问题：

- 配置项是“技术参数式”的，而不是经过设计约束的主题能力
- 预览区直接使用这些值渲染，导致页面很容易呈现出不专业或不一致的视觉效果

换句话说，能力是开放的，但缺少一个“在 Shopify Admin 风格内可控变化”的边界。

##### 6. 配置区分组存在，但卡片语义和层级还不统一

当前左侧配置被拆成多张卡片：

- IP 自动定位
- 选择器类型
- 选择器样式

这已经有一定结构，但问题是：

- 卡片标题都使用默认 `Title level={5}`
- 卡片内部的标签、控件、说明之间没有形成稳定节奏
- 同一页中有的操作是 `Switch`，有的是 `Select`，有的是 `ColorPicker`，但帮助信息与层级表达较弱

因此用户能“操作”，但不一定能快速理解每个配置块的优先级。

##### 7. 引导卡片信息量较重，且状态表达仍依赖手工颜色

`SwitcherSettingCard` 承担了两步启用引导：

- 设置货币格式
- 启用 switcher

这部分业务上是必要的，但当前问题在于：

- 完成/未完成标签使用了手工 RGB 胶囊色
- 引导信息篇幅较长，阅读密度偏高
- “操作按钮 + 状态标签 + 说明文本”没有统一回到系统组件表达

结果是引导卡片像一个独立的信息模块，而不是页面系统的一部分。

##### 8. 预览交互逻辑和页面编辑逻辑耦合较深

当前页面同时维护：

- 配置表单状态
- 保存/取消逻辑
- 预览开关逻辑
- 语言和货币切换模拟逻辑
- 弹层开合逻辑

这些都集中在一个页面文件中，导致：

- 页面职责过重
- 预览区结构很深
- 任何 UI 调整都容易同时影响交互逻辑

这不仅是代码结构问题，也会直接影响后续 UI 重构效率。

##### 9. 样式文件已出现重复和失控迹象

`styles.module.css` 中可以看到明显的问题：

- Google Translate 屏蔽样式重复出现
- 同类样式多次定义
- 既有布局样式，也有组件样式，也有第三方覆盖样式
- 同时混入大量硬编码颜色和交互状态

这说明该页样式体系已经开始失控，不适合继续在现有基础上堆叠。

##### 10. 移动端虽然做了适配，但结构退化较明显

当前移动端会做以下处理：

- 隐藏箭头和部分文案
- 调整预览区与编辑区顺序
- 改造选项列表为底部弹出式布局

这些做法能让页面“可用”，但也暴露出一个问题：  
桌面结构和移动结构之间缺少统一的模式抽象，更多是局部样式修补。

##### 11. 页面中大量局部颜色、边框、背景不来自 token

Switcher 页是当前项目里“颜色来源最分散”的页面之一。  
除了配置值本身，页面实现中还出现了很多写死值，例如：

- `#000000`
- `#ffffff`
- `#ccc`
- `#eee`
- `#f5f5f5`
- `#e8e8e8`
- `#ddd`
- `#888`
- `#555`

这使得它成为最需要做 token 收敛的重点页面之一。

#### 当前结论

Switcher 当前不是单纯的“样式有点乱”，而是：

- 页面模板还没有真正稳定
- 预览区建立了一套独立视觉系统
- 编辑区与预览区的规范来源不一致
- 局部样式和交互逻辑耦合过深

这一页后续重构时，重点应放在：

- 把页面收敛为稳定的双栏编辑模板
- 让预览区回到后台应用风格，而不是独立站演示风格
- 用统一 token 替换散落的硬编码颜色与按钮样式
- 把引导卡片、配置卡片和预览卡片纳入同一套结构规则

### 6.3 Manage Translation

#### 页面当前结构

当前 Manage Translation 页面可以拆为以下几个区域：

1. 页面顶部：`TitleBar + ScrollNotice`
2. 页面头部操作区：语言选择 + Import 入口
3. 主体内容区：按资源类别拆分的多张管理卡片
4. 弹窗反馈区：导入说明弹窗、功能限制弹窗

目前页面已经具备“按资源分类管理翻译内容”的基础结构，但整体更像一组连续堆叠的卡片列表，还没有形成成熟的资源管理页模板。

#### 已确认问题

##### 1. 页面头部仍然不是完整的管理页 Header

当前顶部主要由以下部分组成：

- `TitleBar`
- `ScrollNotice`
- 一个自定义 header 行

其中 header 行只包含：

- `Localized content`
- 语言选择器
- Import 按钮

问题在于：

- 缺少清晰的页面说明
- 缺少对当前管理对象的明确上下文描述
- 没有形成“筛选区 / 操作区 / 结果区”分层明确的后台管理页头部

这与 `design.md` 中资源管理页应具备“头部筛选区 + 批量操作区 + 表格主体”的模板还有距离。

##### 2. 资源管理页当前更像“多张表格卡片堆叠”，而不是一个统一管理界面

页面主体现在由多张 `ManageTranslationsCard` 组成，例如：

- Products
- Online Store Theme
- Online Store
- Blogs and articles
- Images data
- Settings
- Liquid & Third-Party Apps

这种拆分让信息可读，但也带来几个问题：

- 页面纵向很长
- 每张卡片都重复一套表头和动作
- 资源分类之间缺少统一筛选和切换机制
- 用户难以快速建立“这是一个资源管理台”的整体认知

##### 3. 卡片标题层级过重，不符合统一卡片标准

`ManageTranslationsCard` 中卡片标题使用了：

- `fontSize: "1.5rem"`

这明显高于 `design.md` 中推荐的 `Card Title 14px/600`，会导致：

- 每张卡片的视觉权重过高
- 页面出现多个“像大标题一样的卡片标题”
- 真正的页面级层级被稀释

##### 4. 表格已承担主体信息，但表头、密度和操作层级还未统一

当前卡片内部使用 `Table` 展示：

- 资源类型
- 已翻译数量
- 操作按钮

这种结构是合理的，但当前还存在以下问题：

- 所有卡片都重复相同的表格结构
- 表格行和卡片头部的节奏没有统一 token 约束
- 操作列统一是 `Edit`，但缺少更明确的主次操作表达
- 部分资源支持数量统计，部分不支持，当前缺少统一说明方式

##### 5. 状态表达比较弱，资源同步和数量状态不够系统化

当前页面通过以下方式表达状态：

- `Syncing`
- `--`
- 数值 `translated/total`

这虽然能传达基本信息，但还没有形成统一的状态系统，例如：

- 正在同步
- 无统计能力
- 尚无内容
- 已完成
- 部分完成

目前状态更多是“值的显示”，而不是“状态的设计”。

##### 6. 筛选与批量操作能力不足，管理页模板不完整

按 `design.md` 的理想状态，资源管理页应优先具备：

- 头部筛选区
- 批量操作区
- 主体列表区
- 详情抽屉或弹窗

而当前页面只有：

- 语言切换
- Import 按钮
- 分类卡片

这说明它更接近“资源导航页”或“资源入口页”，而不是完整的资源管理页。  
如果后续继续增强功能，这一结构很容易失去一致性。

##### 7. Import 入口和权限限制表达仍然较散

当前 Import 行为包含：

- 免费/低套餐禁用逻辑
- `Popconfirm` 提示升级
- `Feature Unavailable` 弹窗
- 导入说明弹窗

这些功能都可用，但反馈模式比较分散，且入口按钮本身没有和页面主任务形成清晰层级。

##### 8. 页面样式仍然依赖局部绿色状态色和手写 hover 色

`styles.css` 中可见明显的硬编码状态色，例如：

- `#e0f7e0`
- `#29845a`

这些颜色用于选中和 hover 表达，但没有通过 Polaris token 派生，仍然属于局部手工配色。

##### 9. 页面结构和数据请求高度耦合，影响后续 UI 调整

当前页面里存在大量：

- `useFetcher`
- 重复请求块
- 多段 `useEffect`
- 多组数据源定义

这会带来一个直接问题：  
当后续要调整页面结构、卡片分组或表格模板时，UI 改动会很容易牵动请求逻辑和状态更新逻辑，增加重构风险。

##### 10. 子组件 `ItemsScroll` 暴露出实现质量不稳定的问题

当前 `itemsScroll.tsx` 中可以看到明显异常内容，例如直接渲染了一大段固定字符串，而不是 `item.label`。  
无论它当前是否仍在主路径使用，这都说明该页面相关组件存在以下问题：

- 局部组件缺少统一治理
- 一些 UI 组件并没有保持稳定质量
- 页面周边实现已经出现“能工作但不够收敛”的迹象

##### 11. 页面宽度和管理密度还有继续优化空间

`design.md` 明确建议资源管理页优先使用 full-width 布局。  
当前页面虽然已经偏宽，但主体仍以多张卡片纵向堆叠为主，视觉上更像普通内容页，而不是高密度后台管理页。

这意味着后续应继续增强：

- 头部筛选的清晰度
- 列表区的信息密度
- 分类切换与结果区之间的结构关系

#### 当前结论

Manage Translation 当前的核心问题不是功能缺失，而是页面角色还不够稳定。  
它现在更像“翻译资源入口集合页”，而不是一套成熟的资源管理页。

当前最主要的 UI / 交互问题集中在：

- 页面头部管理语义不足
- 分类卡片过多，页面纵向压力较大
- 卡片标题层级过重
- 表格状态表达和密度规则未统一
- 筛选区、操作区、结果区还没有真正形成管理页模板

后续重构时，这一页更适合沿着“资源管理台”方向继续收敛，而不是继续给每个资源分类单独叠加更多卡片和零散交互。

#### 页面当前结构

当前 Pricing 页面可以拆为以下几个区域：

5. 套餐对比区：计划能力表格

7. 辅助弹窗区：取消套餐、Shared Plan、免费试用提示等

页面信息已经很完整，但视觉表达和交互结构明显更接近“营销式 pricing page”，而不是 Shopify Admin 内部的计费与套餐管理页。

#### 已确认问题

##### 1. 页面整体气质偏营销页，不够像后台应用计费页

根据 `design.md`，Pricing 页应重点避免：

- 大面积强色块
- 营销着陆页式表达
- 夸张字号
- 强刺激式推荐区

而当前页面仍然有明显的营销页倾向，例如：

- 计划卡片强调感过强
- 推荐标签使用高饱和蓝色 ribbon
- 有滚动 marquee 文案
- 套餐区和 FAQ 区更像独立落地页模块

这使得页面整体偏离了“后台应用内冷静、可信的计费管理页面”气质。

##### 2. 顶部额度区存在独立强视觉，不符合统一后台卡片风格

`AcountInfoCard` 当前做法包括：

- 单独设置 `Card.headerBg = #007F61`
- 顶部跑马灯文案
- 强色头部和白字信息

这实际上又建立了一套页面局部视觉系统，且与其他页面卡片风格差异明显。  
从后台管理视角看，这个区域的信息应该清晰可信，而不是以“高强调 banner 卡片”方式出现。

##### 3. 颜色来源仍高度分散，存在大量硬编码品牌色和状态色

Pricing 页中反复出现如下颜色：

- `#007F61`
- `#1890ff`
- `#52c41a`
- `rgb(255, 230, 0)`
- `rgb(254, 247, 246)`
- `rgb(253, 223, 217)`

这些颜色既来自旧的品牌表达，也来自 Ant Design 默认色，还混入了局部营销色块，没有统一回到 Polaris token。

##### 4. 字号层级偏大，价格区有明显夸张化表达

当前页面多处出现较激进的文字层级，例如：

- 免费套餐价格 `28px`
- 套餐卡片价格 `28px`
- 模态标题 `24px`
- 某些卡片标题 `20px`
- `Manage / Compare / FAQ` 这些区块标题也偏大

这会让页面更像营销销售页，而不是 Shopify App 后台中的套餐管理页。

##### 5. 套餐信息分层还不够稳定

`design.md` 明确要求 Pricing 页面中以下信息要明确分层：

- 当前计划
- 推荐计划
- 购买额度

而当前页面里这些内容虽然都有，但层级关系并不够稳定：

- 页面顶部显示当前计划
- 套餐区又重复强调计划状态
- Buy Credits 独立成另一组购买逻辑
- Shared Plan 又作为额外付费概念插入其中

结果是信息都在，但页面主线不够清晰。

##### 6. 套餐卡片的视觉表达过重，且卡片之间存在多套强调规则

当前套餐卡片用多种方式表达差异：

- 当前计划边框高亮
- Recommended 蓝色 Ribbon
- 免费试用按钮
- 不同按钮样式
- 不同价格说明区块

这导致用户会同时看到多个“重点”，而不是一个清晰的推荐逻辑。

##### 7. “Buy Credits” 区块与套餐区块风格不统一

`BuyCreditsOuterCard` 当前更像一个快捷操作条，而主页面中的 Buy Credits Modal 又是另一套卡片选择界面。  
问题在于：

- 顶部加购额度区和下方套餐区没有统一视觉语言
- 按钮、价格、选中态都在用局部颜色和手工边框
- 购买“套餐”和购买“额度”是两套重要动作，但页面没有很好地组织它们之间的关系

##### 8. 对比表格信息很全，但密度和可读性还可继续收敛

套餐对比表具备完整内容，这是优点。  
但当前问题在于：

- 表格是典型功能对比表结构，更接近营销官网样式
- 文案长度差异大，可读性压力较高
- 当前列标题和单元格风格未充分收敛到后台应用表格规范

换句话说，它“能看”，但还不够像后台应用内部的标准对比区。

##### 9. FAQ 区块延续了落地页式结构

页面底部使用：

- 左侧标题与说明
- 右侧折叠问答

这是典型营销页/官网 FAQ 模块的结构。  
在后台应用场景中，这种表达不是不能用，但当前它的存在进一步强化了页面的“落地页感”，而不是产品内部的“帮助补充区”。

##### 10. 多个弹窗的视觉和信息强度不一致

当前 Pricing 页涉及多个弹窗：

- Buy Credits
- Cancel paid plan
- Shared Plan
- 免费试用提示

它们分别使用了不同强度的标题、卡片、说明块和按钮表达。  
这会带来两个问题：

- 模态层级和信息密度缺少统一标准
- 一些弹窗明显比页面正文更“营销化”或更“强调式”

##### 11. 部分辅助视觉元素不够专业

当前页面中一些辅助视觉表达较容易削弱专业感，例如：

- 跑马灯文案
- 黄色“Save 20%”胶囊
- 蓝色 Recommended Ribbon
- 大面积渐变风格免费试用提示样式

这些元素虽然能增强促销感，但不符合 `design.md` 中“Pricing 页面应回归冷静、可信、后台应用风格”的要求。

#### 当前结论

Pricing 当前最大的问题不是“内容不完整”，而是“表达方式不够克制”。  
它已经包含了套餐页所需的主要信息，但很多视觉与交互表达仍然带有明显的营销页特征。

这一页后续重构时，重点应放在：

- 把页面从“促销式 pricing”收敛为“后台式 billing”
- 降低强色块、跑马灯、夸张字号和强刺激标签的使用
- 明确当前计划、推荐计划、额度购买三者的层级关系
- 用更统一的卡片、表格和 modal 规范替代局部营销式设计

这类页面整体都已经具备“配置页 / 管理页”雏形，但仍存在结构、交互和视觉规范不一致的问题。

#### Language

##### 页面当前结构

当前 Language 页面主要包含：

- 桌面端和移动端对 `Publish` 状态的表达逻辑不完全一致，两个端的状态语义存在偏差。
- `Publish` 使用的是 `Switch` 外观，但实际行为是打开弹窗处理，控件预期和真实行为不一致。
- 页面初始化依赖多次请求与轮询，状态刷新链路偏复杂，容易造成加载抖动与状态延迟。
- 高风险动作的确认策略不统一，删除有单独提醒逻辑，但发布和自动翻译等重要动作没有同等级的风险控制。
- 页面整体已经很接近管理页，但头部说明、列表状态、批量动作和反馈模式仍未形成统一模板。

##### 当前结论

Language 页面当前最核心的问题不是功能缺失，而是管理规则和交互语义还不稳定。  
它需要进一步收敛为一套更统一的“语言管理页模板”。

#### Currency

##### 页面当前结构

当前 Currency 页面主要包含：

1. 默认货币说明与顶部操作区
2. 桌面端货币表格
3. 移动端卡片列表与分页
4. 新增货币与编辑汇率弹窗

##### 已确认问题

- 桌面端与移动端的数据来源并不完全一致，后续恢复搜索或过滤后可能出现两端结果不一致。
- 移动端自动汇率读取逻辑存在实现问题，展示值可能异常或为空。
- 删除操作缺少充分确认，尤其在批量删除场景下误操作风险偏高。
- 页面原本显然需要搜索，但当前搜索区被注释或弱化，数据量增大后可用性会下降。
- 页面结构上仍更像普通列表页，而不是一个“货币设置管理页”，信息层级和风险动作表达都偏轻。

##### 当前结论

Currency 页面是标准设置列表页，但需要补齐搜索、确认反馈和跨端一致性，才能真正纳入统一设置页体系。

#### ApiKeySetting

##### 页面当前结构

当前 API Key 设置页主要包含：

1. `TitleBar` 与页面内返回按钮
2. 两张 API 配置卡片
3. 配置弹窗
4. 测试翻译弹窗

##### 已确认问题

- 顶部同时存在 TitleBar breadcrumb 和页面内返回箭头，导航层级重复。
- 输入校验主要发生在提交时，缺少更明确的字段级即时反馈。
- 测试结果区大量依赖原生输入区域，风格与页面其他部分不一致。
- API 卡片在数据未稳定时仍可触发操作，更多依赖 toast 提示，不够直观。
- 页面虽已按“卡片 + 弹窗”拆分，但表单帮助、错误提示和结果反馈还没完全系统化。

##### 当前结论

ApiKeySetting 页面在结构上已经可用，但离高质量设置页仍差在“输入反馈、导航一致性、测试结果展示规范化”这三个点。

#### Glossary

##### 页面当前结构

当前 Glossary 页面主要包含：

1. 顶部标题与说明
2. 术语表列表区
3. 桌面端表格 / 移动端卡片
4. 状态开关、编辑、批量删除
5. 创建规则与升级限制弹窗

##### 已确认问题

- 套餐限制同时通过按钮置灰、hover 提示、升级弹窗等多种模式表达，策略不统一。
- `Status` 开关是异步写接口，但失败时主要靠 toast，缺少更明确的状态回滚和字段反馈。
- 已选数量与辅助文案的可读性细节不足，说明页面信息层级还没完全打磨到位。
- 删除术语规则缺少足够强的确认流程，批量删除风险偏高。
- 页面具备基础管理页结构，但状态系统、限制反馈与批量操作规范还不够稳定。

##### 当前结论

Glossary 页面已经接近成熟管理页，但需要进一步统一“状态开关、套餐限制、批量删除、反馈机制”的交互模式。

- 日期筛选更接近前端裁剪，不是稳定的数据筛选模型。

##### 页面当前结构

- 吸顶头部依赖手工灰底和 `Affix`，与整体嵌入式页面体系不够一致。
- 页面内部存在多组独立 loading 状态，但区域间反馈并不总是同步。

##### 当前结论

3. 模块选择区
4. 任务列表区
5. 固定客服组件

- 活跃任务轮询频繁，任务一多时会带来明显刷新和性能压力。
- 模块选择区缺少分组、搜索和说明，扫读成本偏高。
- 右下角客服组件是完全自定义 fixed 布局，在小屏和可访问性方面存在风险。

##### 当前结论

translate-v4 已经是高能力密度页面，但需要从“功能堆叠页”收敛为更清晰的任务创建与任务管理页面。

### 6.8 资源详情与专项模板页补充审查

本组页面包括：

- `app.manage_translation_*` 全部资源详情页
- `app.switcher_.custom_redirects`

这部分页面数量很多，但从 UI 结构上看，可归并为以下几种模板。

#### 模板 A：标准资源编辑页

代表页面：

- `app.manage_translation_.page/route.tsx`
- `app.manage_translation_.article/route.tsx`
- `app.manage_translation_.blog/route.tsx`
- `app.manage_translation_.collection/route.tsx`
- `app.manage_translation_.delivery/route.tsx`
- `app.manage_translation_.email/route.tsx`
- `app.manage_translation_.filter/route.tsx`
- `app.manage_translation_.json_template/route.tsx`
- `app.manage_translation_.locale_content/route.tsx`
- `app.manage_translation_.metafield/route.tsx`
- `app.manage_translation_.metaobject/route.tsx`
- `app.manage_translation_.navigation/route.tsx`
- `app.manage_translation_.policy/route.tsx`
- `app.manage_translation_.section_group/route.tsx`
- `app.manage_translation_.settings_category/route.tsx`
- `app.manage_translation_.settings_data_sections/route.tsx`
- `app.manage_translation_.shipping/route.tsx`
- `app.manage_translation_.shop/route.tsx`

##### 共有问题

- 页面骨架通常是 `SaveBar + SideMenu + 内容区`，方向正确，但模板抽象不足。
- 页面头部普遍只有标题和返回，没有充分的说明、筛选与状态区。
- Polaris 与 Ant Design 混用严重，骨架和内部控件不属于同一套设计语言。
- 桌面端与移动端普遍写成两套渲染结构，重复度高。
- 内联样式和局部布局逻辑较多，影响后续统一收敛。
- 单文件体量过大，UI 和状态逻辑耦合严重。

##### 当前结论

标准资源编辑页已经有统一雏形，但需要被真正提炼成一个稳定模板，而不是继续复制 route 文件。

#### 模板 B：产品与媒体类资源页

代表页面：

- `app.manage_translation_.product/route.tsx`
- `app.manage_translation_.productImage/route.tsx`
- `app.manage_translation_.productImageAlt/route.tsx`

##### 共有问题

- 除了基础编辑结构，还叠加了产品分页、子资源分页、预览和保存逻辑，复杂度高。
- 页面在“资源浏览”和“资源编辑”之间来回切换，心智负担明显高于普通资源页。
- 单页承担了过多职责，任何 UI 调整都容易牵动交互结构。

##### 当前结论

这组页面不应只被视为“标准模板的一个分支”，而应视为更高复杂度的媒体编辑页类型。

#### 模板 C：规则表格 CRUD 页

代表页面：

- `app.manage_translation_.custom_liquid/route.tsx`
- `app.switcher_.custom_redirects/route.tsx`

##### 共有问题

- 页面主体围绕表格与弹窗展开，但缺少统一的保存、确认和批量动作规范。
- 桌面端表格与移动端卡片双分支重复较多。
- 初始化请求与状态联动较重，页面结构脆弱。

##### 特殊问题

- `custom_liquid` 存在字段标签与实际内容映射错误，说明页面实现稳定性不足。
- `custom_redirects` 没有采用统一 SaveBar 编辑体验，和其他编辑页割裂。

##### 当前结论

这类页面应被单独定义为“规则管理页模板”，而不是继续作为局部业务页面各自实现。

#### 模板 D：主题文件 / PageFly 专项页

代表页面：

- `app.manage_translation_.pagefly/route.tsx`

##### 已确认问题

- 资源发现依赖文件名与约定规则，鲁棒性偏弱。
- 页面以大文本块编辑为主，字段级结构不足。
- 视觉结构和其他资源编辑页差异很大，但又没有形成独立的专项模板。

##### 当前结论

PageFly 页面属于高特殊性页面，后续应单独定义“长文本 / 主题文件编辑页”的结构标准。
